Security policy

Consolidated security and privacy policy. Reflects locked decisions D-001 through D-011 in narrative form for review. Read together with the privacy policy and the pre-install checklist.

1. Threat model

YesAndBudget runs as a single-user desktop application on the consumer's own device. The threats it protects against:

  • Unauthorized network exfiltration of bank data. Addressed by the no-cloud architecture (D-001): there is no operator-controlled server, no shared backend, no telemetry pipeline. The application's API binds to 127.0.0.1 only (D-006) and refuses connections from any external interface.
  • Credential leakage from version control. Addressed by gitignoring every credential-bearing file (.plaid-token, data/, .env*) and by static review of every release.
  • Accidental data corruption during import. Addressed by pre-import snapshots and the rollback endpoint that walks every transaction inserted by a batch (D-004, D-007).
  • Confused-deputy bugs against the user's data. Addressed by the D-011 archive-before-delete pattern: switching an account between CSV and Plaid modes archives the displaced rows, never deletes.

The threats it does NOT defend against:

  • Compromise of the consumer's operating system. If the OS is compromised, all data on the device is recoverable by the attacker. Defenses against this are at the OS layer: full-disk encryption, OS login, and patch management.
  • Theft of an unlocked device. Same as above. Lock your device when unattended.
  • Insider threat from someone with shared access to the consumer's user session. Standard desktop-app threat model.

None of this data is transmitted to any operator-controlled server. The only outbound network traffic is to production.plaid.com (HTTPS, TLS 1.2+) under the consumer's Plaid credentials.

3. Access control

There is no shared infrastructure. Access control is delegated to the OS. The application's HTTP server binds to 127.0.0.1 only (D-006). The application does not impose a separate authentication layer on top of OS login by default; the optional in-app passphrase lock (Settings → App lock) provides an additional UI-layer gate when enabled.

4. Cryptography

In transit: TLS 1.2+ for all outbound traffic to Plaid. Localhost loopback traffic is plain HTTP because it never traverses any network outside the device.

At rest: OS-layer full-disk encryption (BitLocker / FileVault recommended; see the pre-install checklist). Application-layer encryption via SQLCipher is documented as a future hardening step.

5. Consent

Captured at two layers:

  1. Plaid Link consent — when the consumer initiates a bank connection, Plaid's own consent screen describes the data being collected and obtains explicit consent before the link completes.
  2. YesAndBudget takeover confirmation — when a Plaid link affects existing CSV-imported data, the application presents a per-account preview showing exactly which prior records will be archived. The consumer must explicitly confirm before any change is made. See D-011.

A first-launch data-use modal presents the consumer with a summary of what the application stores, where, and how to delete it. Acceptance is recorded with a timestamp and the application version.

6. Deletion and retention

The consumer fully controls retention. Per-transaction deletion, per-import rollback, per-Plaid-link unlink, and a Settings "Delete all my data" factory reset are all exposed in the UI. There is no operator-side copy that survives because there is no operator side.

7. Vulnerability management

Dependency vulnerabilities are scanned on every pull request via Dependabot and a pnpm audit CI gate. High-severity findings block the release script. Endpoint hardening on each user's installation relies on the platform's built-in tooling: Windows Defender / equivalent endpoint protection, OS-level patch management, and dependency vulnerability scanning.

8. Incident response

If a vulnerability is reported and confirmed, the fix ships in the next release with a CHANGELOG entry. If consumer data exposure is possible, a notice is posted at security-notices listing affected versions and the recommended user action. If a stored credential has been exposed, rotate it via the relevant dashboard and update .plaid-token accordingly.

9. Review

This policy is reviewed on every release. The locked decisions it consolidates (D-001 through D-011) require an ADR-promoter call to change.
